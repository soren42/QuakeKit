1. Purpose

This styleguide defines how QuakeKit documentation should read, look, print, and age. It exists so the User Manual, Developer Guide, Plugin Developer Guide, and Theme Designer Handbook feel like one documentation system rather than four unrelated files.

The project is still moving quickly. These documents should be complete enough to orient a reader today, explicit about what is still alpha-quality, and easy to update when the first signed .app bundle replaces command-line launch testing.

RuleDocumentation must tell the truth about the current implementation. When a feature is planned but not done, describe the intent and mark the gap clearly.

3. Page Model

Manuals are HTML documents designed for both direct browser reading and US Letter PDF output. Use the shared stylesheet at docs/manuals/assets/manual.css. The default printed page target is:

  • Paper: US Letter, portrait.
  • Print margin: 0.55in.
  • Screen body width: 8.5in maximum.
  • Cover page: large title, metadata, brief scope.
  • Tables and figures: avoid page breaks inside whenever possible.

Every manual should use this base structure:

<!doctype html>
<html lang="en">
<head>
  <meta charset="utf-8">
  <title>QuakeKit Manual Title</title>
  <link rel="stylesheet" href="assets/manual.css">
  <script src="assets/manual.js" defer></script>
</head>
<body>
<main class="manual">
  <section class="cover">...</section>
  <section><h2>Table of Contents</h2>...</section>
  ...
</main>
</body>
</html>

4. Voice And Terminology

QuakeKit documentation should be direct and helpful. It should not sound like marketing copy. Explain what a reader can do, what can go wrong, and what to check next.

UseAvoidReason
QuakeKitOpenQuake, MacQuake, DK appQuakeKit is the project name moving forward.
DK-QUAKE displayscreen gadget, small monitorNames the physical target precisely.
Functional pluginplugin, app, widget interchangeablySeparates behavior from theme packages.
Theme packageskin, CSS packThemes define tokens, assets, components, and hardware presentation.
Menu bar accessorysystem traymacOS uses menu bar terminology. “System tray” can be mentioned for Windows-oriented readers.
Current alpha buildrelease-quality replacementThe current implementation is still pre-v1.0.

5. Structure Standards

Headings

Use numbered top-level sections in manuals. Keep headings short and literal. Prefer “Install QuakeKit” over “Getting Everything Ready.”

Task Sections

User-facing task sections should follow this order: what the task accomplishes, prerequisites, steps, expected result, troubleshooting note. Developer-facing task sections should add verification commands.

Known Gaps

If a section describes a future release behavior, include a gap note near the relevant text rather than burying it at the end.

GapThe final QuakeKit release is expected to ship as a signed .app bundle. Current development builds can assemble a local bundle, but packaging, signing, notarization, and first-run UX still need final release work.

6. Visual Standards

Use diagrams when they compress architecture or workflow complexity. Use screenshots only when they reveal concrete UI state that prose cannot describe efficiently. Because the app is changing quickly, early manuals should prefer SVG diagrams and schematic UI figures over brittle screenshots.

Standard architecture figure. Reuse this diagram across manuals when explaining the runtime boundary.

Figure captions should explain why the figure matters. Do not caption with only “Architecture diagram” or “Screenshot.”

7. Code And Commands

Commands should be copyable, minimal, and run from the repository root unless the text says otherwise. Prefer full command blocks over inline command chains when the command is a task the reader might execute.

cd /path/to/QuakeKit
./scripts/validate-release.sh

For JSON examples, include enough context for validation. Avoid fragments unless the text clearly says “excerpt.”

{
  "$schema": "../../schemas/functional-plugin.schema.json",
  "id": "example_status",
  "name": "Example Status",
  "version": "1.0.0",
  "apiVersion": "0.1",
  "entry": { "transport": "shell", "command": "status.sh" }
}

8. Reference Tables

Reference tables should use consistent columns: field, type, required, purpose. If a table documents user-visible controls, use setting, default, effect, restart required.

FieldTypeRequiredPurpose
idstringYesStable lowercase identifier for package discovery and stored preferences.
versionstringYesHuman-readable package version. SemVer is recommended.

9. Gaps And Draft Markers

Use one of three explicit markers:

MarkerUse WhenExample
AlphaFeature exists but behavior, UI, or packaging may change.Current plugin settings surface.
GapFeature is intended but incomplete.Signed/notarized installer.
RiskFeature touches permissions, data privacy, hardware safety, or third-party terms.Microphone recording, media automation, AI harnesses.

10. Accessibility

  • Every image must include meaningful alt text.
  • Do not encode status only with color. Pair color with labels or symbols.
  • Use real headings in order; do not fake headings with bold paragraphs.
  • Tables must have header rows.
  • Code blocks must preserve whitespace and avoid horizontal-only scrolling in print.

11. Release Checklist

  1. Open each manual in Safari or Chromium and verify no missing local assets.
  2. Print each manual to PDF on US Letter and inspect cover, table breaks, code wrapping, and diagram scaling.
  3. Run ./scripts/validate-release.sh before publishing manuals with release artifacts.
  4. Update all “current alpha” notes when the signed .app bundle is ready.
  5. Refresh plugin and theme inventory tables from the current repository before tagging v1.0.
