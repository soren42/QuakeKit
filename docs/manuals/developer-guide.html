1. Architecture

QuakeKit is intentionally a platform, not a single-purpose dashboard. The macOS DK-QUAKE host is the first host implementation. The runtime and plugin contracts should remain portable enough for future Windows and Linux hosts.

QuakeKit separates device I/O, runtime state, plugin contracts, host UI, and third-party extension packages.

LayerResponsibilityMust Avoid
Device adaptersTransport I/O, byte framing, safe commands, normalized events.Application layout, plugin policy, UI assumptions.
Runtime corePages, widgets, actions, data snapshots, event bus, knob-ring arbitration.Hard dependency on AppKit or DK-specific display geometry.
Plugin APICodable manifests, settings, permissions, messages, package validation.Executing untrusted code directly inside host process.
Host appmacOS windows, settings, menu bar, audio, package install, permissions, display ownership.Embedding business logic that belongs in plugins.

3. Build And Validation

Use Xcode’s developer directory when CommandLineTools lacks the Swift Testing module:

export DEVELOPER_DIR=/Applications/Xcode.app/Contents/Developer

Core workflow:

swift build
swift test
swift run quake-test
./scripts/validate-release.sh

scripts/validate-release.sh runs the full non-hardware gate: Swift build/test, quake-test, plugin manifest validation, theme manifest validation, adapter JSON validation, and local app bundle assembly.

4. macOS Host

The AppKit host launches as an accessory app by default. It creates a menu-bar status item, opens a high-level borderless window on the DK display, starts a pointer guard, acquires a display sleep assertion, and starts HID unless --no-hid is used.

FlagUse
--foregroundRegular app activation for debugging.
--debug-windowResizable non-fullscreen host window.
--main-screenRender on the main display.
--no-hidDisplay-only mode.
--shared-hidShared HID open mode for debugging alongside other software.
--strict-hid-seizeFail HID startup instead of falling back if seize fails.

--keepalive vendor|screenOn|aggressiveKeep-alive behavior for diagnostics.

5. Runtime Model

The runtime normalizes HID input and plugin output into UI-facing state. Important concepts are:

  • RuntimeEvent: connected, touch, knob, state, action, and stream events.
  • PluginDataStore: captures action results and data snapshots for display.
  • PluginExecutionHost: executes supported process-style plugin actions.
  • KnobRingCoordinator: arbitrates semantic LED-ring requests by source, priority, TTL, and theme mapping.

BoundaryFunctional plugins request semantic states such as success, warning, danger, focus, or progress. Themes map those states to colors and animations.

6. Package Discovery

Bundled packages load from Examples/Plugins and Examples/Themes. Installed packages load from Application Support and override bundled packages with the same manifest ID.

~/Library/Application Support/QuakeKit/Plugins/
~/Library/Application Support/QuakeKit/Themes/

Accepted install inputs are unpacked .quakekitplugin and .quakekittheme directories, plus .tar, .tar.gz, and .tgz archives containing one package directory.

7. Hardware Safety

The DK control protocol silently drops malformed frames, so protocol tests pin exact vendor command bytes. Do not add exploratory write commands to production paths. Keep unknown protocol experimentation isolated, reviewed, and opt-in.

RiskThe known DFU command must never be used as a wake probe or generic diagnostic. Keep wake behavior limited to known-safe screen-on, ping, query, brightness, microphone, and knob-ring commands.

9. Release Checklist

  1. Run ./scripts/validate-release.sh.
  2. Verify clean git status.
  3. Build local app bundle and launch with DK hardware connected.
  4. Confirm panel wake, touch, knob, LED ring, settings window, package install/remove, and carousel behavior.
  5. Print manuals to PDF and verify layout.
  6. For public release, sign/notarize the .app and update install sections.

10. Engineering Gaps

  • JSON Schema files are normative, but CLI validation currently uses Swift decoding/validators rather than executing JSON Schema constraints directly.
  • shell, stdioJSONRPC, and php action execution are implemented; websocket, webView, and nativeSwift are contract surfaces with incomplete execution support.
  • Permission prompts, grant persistence, and Keychain-backed secret storage need production implementation.
  • Theme asset existence and richer web applet lifecycle validation need hardening.
  • Final signed app packaging, updater strategy, and crash/log collection are pending.
